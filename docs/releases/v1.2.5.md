# v1.2.5 更新说明

本版本继续修复 WebUI 更新流程和外部工具管理体验，重点解决更新后页面停留、Tools 页面必需工具卸载提示不明确、版本号展示不醒目等问题。

## 更新与登录流程

- 点击顶部“立即更新”后，现在会显示明确的提交、等待重启、完成刷新状态，不再出现点完没有任何反馈的情况。
- 程序更新完成后会自动轮询 `/api/info` 判断服务是否恢复；服务恢复后自动刷新页面。
- 如果更新后会话失效或需要重新登录，会自动跳转到登录页，并保留当前访问路径。
- 修复更新页面在无录制任务时仍被当成“优雅等待更新”的判断问题，只有后端明确返回等待状态时才显示等待更新。
- API 错误对象现在保留 HTTP 状态码，前端可以准确识别 `401/403` 并跳转登录。

## 版本号显示

- 主 WebUI 左下角增加当前版本号常驻显示。
- 登录页左下角增加当前版本号显示。
- Tools、Scheduler 等外部代理页面也会注入当前版本号显示，便于确认当前运行的是哪个版本。

## 外部工具管理

- 必需 / 默认安装工具只剩最后一个已安装版本时，禁止卸载所有版本，并返回明确说明：
  “这是必需工具，至少要保留一个已安装版本，不能卸载所有版本；如需切换，先安装其他版本再卸载当前版本。”
- 修复外部 Tools 服务返回空 `409` 时，页面只能显示“卸载失败”的问题。现在 BiliLive-go 会补齐 `err_msg`、`message` 和 `data.reason`，前端可以显示具体原因。
- 兼容 Tools API 的真实返回结构 `{"value":[...]}`，避免必需工具状态判断失败。
- 兼容工具操作请求里的 `toolName`、`name`、`tool` 字段，减少不同 Tools 页面版本之间的请求字段差异导致的漏判。
- 为 `/tools/api/uninstall` 和 `/tools/api/toggle` 增加显式保护路由，确保这些危险操作不会绕过 BiliLive-go 的保护逻辑。
- Tools 页面注入脚本增加 fetch / XHR 错误兜底提示，外部页面吞掉错误时仍尽量显示明确原因。
