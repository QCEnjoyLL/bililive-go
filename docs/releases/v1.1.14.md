# v1.1.14 更新说明

本版本继续修复 Docker 首次启动、WebUI 登录体验和 BoyFriend HLS 录制诊断问题。建议正在使用 `v1.1.13` 的 Docker 用户升级测试。

## WebUI 登录体验

- 将浏览器自带的 Basic Auth 弹窗改为独立 `/login` 登录页。
- 登录成功后使用服务端 `HttpOnly` 会话 Cookie，浏览器页面脚本无法直接读取会话令牌。
- 未登录访问页面会自动跳转登录页，未登录访问 API 会返回 JSON 401。
- 保留 Basic Auth 作为脚本或接口调用的兼容入口。

## Docker 首次录制准备

- 修复 Docker 新镜像首次启动后，WebUI 已可用但 FFmpeg 尚未下载完成时，立即添加房间会录制失败的问题。
- 录制启动前会主动确保 FFmpeg 可用；如果 FFmpeg 正在下载或尚未安装，会等待或触发 RemoteTools 准备完成。
- HLS、普通 FFmpeg、WebRTC、browser 录制路径都接入同一套 FFmpeg 准备逻辑。
- WebUI 新增录制环境状态提示，能看到 FFmpeg、bililive-tools、录播姬、调度器等组件是否已准备好。

## HLS 录制诊断与稳定性

- HLS playlist 请求增加防缓存参数，并带上 `Cache-Control: no-cache` / `Pragma: no-cache`，降低 CDN 返回过短或滞后 playlist 导致漏看分段的概率。
- 定向 `_HLS_msn` playlist 失败后的回退时间从 10 分钟缩短到 1 分钟，让程序能更频繁尝试恢复定向补洞。
- 诊断日志新增 `疑似漏看本周期/累计`，避免 `确认丢段 0` 掩盖 playlist 滑动窗口漏段问题。
- 临时新增 `hlsmouflon 监视` 日志，将覆盖率、窗口大小、liveLag、下载耗时、写入等待和风险原因汇总成一行，方便继续定位卡顿来源。

## 测试建议

- 如果仍遇到卡顿，请重点贴 `hlsmouflon 监视` 日志行。
- 重点观察 `覆盖率` 是否接近 100%、`风险` 是否仍提示 `playlist滑窗漏段` 或 `窗口过短`。
- 如果 FFmpeg 首次准备较慢，请等待 WebUI 顶部录制环境提示变为可用后再开始录制。
